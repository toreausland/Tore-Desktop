Lessons Learned — Applied Across All Tiers

  1. XML-level zipfile fixes are the most reliable approach for targeted changes (color, font, text swaps). They preserve all existing content, images, and formatting. Template rebuilds are better for heavily restructured files.
  2. Scan the entire document for all instances of any issue — not just the specific location flagged in reports.
  3. Check for retired document names (Markedsanalyse Master, Market Analysis Master, DRAFT_03) — these propagate from source content.
  4. Check for version numbers in document references (V3, V2, etc.) — only VDR reference numbers are allowed per Design Manual §13.6.
  5. Verify chapter/section cross-references still make sense when the target document has been replaced by a new document with different structure.
  6. Check for banned Norwegian terms (Konfidensielt, Vedlegg, Komplett) and translate to English.
  7. Handle split XML runs — Word often splits text across multiple runs; simple string search on individual runs misses cross-run patterns like “REV” + “6”.
  8. Always copy to temp before reading to avoid file lock errors when user has file open in Word.
  9. Use glob for path resolution to handle Norwegian characters in OneDrive paths.

6. Methodology & Tools

Four complementary approaches used across all tiers, proven reliable through 36 .docx file completions + 1 PPTX presentation.

Quad Approach: Template Rebuild + XML Surgery + PPTX Build + Content Creation

Approach 1 — Template Rebuild (aurelian_docx_template.py / python-docx): Used for files requiring heavy restructuring or translation. Produces guaranteed 9/9 output with consistent styling. Used for: Use of Funds, Founder Contribution, CAPEX, TAM/SAM/SOM, Concept Note, IP Assignment, Investment Teaser, Org Chart, Model Assumptions, Economic Tables REV7.

Approach 2 — XML-Level Fix (zipfile module): Used for surgical corrections — color swaps (#FF3E00→#F50537), font changes (Arial→Calibri), text replacements (REV6, banned names). Preserves all existing content, images, tables, and formatting. Used for: Shareholders Agreement, Acceleration Terms, CNC Equipment, Tore CV, André CV, DD Checklist, Executive Summary.

Approach 3 — PPTX Build (python-pptx): Used for presentations. Built Investor Pitch Deck V4 (14 slides) from scratch with Design Manual v2.0 presentation rules.

Approach 4 — Content Creation (.md source files): Used for new document creation from scratch. Content written as Markdown, verified against master document (17 checkpoints), reviewed/revised by user. Conversion to .docx via Cowork with Design Manual v2.0 specifications. Used for: Company Fact Sheet, Henrik CV, Advisor Bios, Employment Agreement, Advisory Agreement, MAZAK LOI, Customer LOI, NDA.

Approach 5 — Cap Table Cascade (python-docx + openpyxl + python-pptx): Used for multi-document co-founder integration. Systematic 7-phase update across .docx, .xlsx, .pptx, .md, and .json files. Table row insertion with deepcopy styling. Red thread verification scan across 94 files. Used for: REV8 co-founder integration (24+ files).

Key insight: XML-level fixes are instant and reliable for targeted changes. Template rebuilds are better when the file structure needs significant modification. The .md-first approach for new documents allows rapid content iteration with user, followed by batch .docx conversion. All approaches produce verifiable results with automated checkpoint verification.
